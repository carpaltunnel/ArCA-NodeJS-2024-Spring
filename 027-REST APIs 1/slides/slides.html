REST APIs





REpresentational State Transfer style Application Programming Interfaces. Or, a standardized and easy way to understand and interact with a webservice...

Principles


  • Client-Server - HTTP, separate the UI and API
  • Stateless - HTTP, every request stands alone
  • Cacheable
  • Uniform Interface (contract)
  • Layered Architecture (abstraction)

Resources


It's safe to assume every "thing" is a resource. If you have an API that manages widgets, widgets are your resource. Each specific instance of a resource should have a unique identifier.

Methods / Verbs

  • GET - Request resource. This could be an individual piece of data or a list of data (resources). EG : Give me a list of widgets.
  • POST - Create resource. A POST indicates that a request is asking to create a new resource. EG : Create a widget for me, here are its properties.
  • PUT - Update an existing resource. PUT indicates that a request wishes to update a resource with a full replace. EG : I don't care what widget #27's old properties were, here is what they should be now.
  • DELETE - Delete an existing resource. EG : I don't need widget #83 anymore, delete it.
  • PATCH - Update part of an existing resource. EG : Change the description of widget #27 from X to Y, leave the rest of it as is.

Routes

Routes are the standardized URLs in an API that clients will use to interact with it. They should be named appropriately to describe the resource they support.

  • API Version
  • Base (Resource)
  • Inline Parameters
  • Query Parameters

Route / URL Examples

  • GET https://.../api/v1/widgets
  • *POST https://.../api/v1/widgets
  • GET https://.../api/v1/widgets/27
  • *PUT https://.../api/v1/widgets/27
  • *PATCH https://.../api/v1/widgets/27
  • (*?)DELETE https://.../api/v1/widgets/27
* Indicates a request body.

JSON

We've talked about how JSON is structured and how to serialize/deserialize it in Javascript. While it is not the only way to transfer data over the internet, it's probably the most common. For all of the APIs we interact with and build, we'll use JSON as our transport language.
