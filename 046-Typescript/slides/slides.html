Typescript





Why Typescript?

Similar to how ES6 fixed a lot of the inherent nastiness of Javascript, Typescript's purpose is to go further and make it more difficult to make logical errors that can wreak havoc on your application.

  • Simple / Primitive Types
  • Parameter Types
  • Return Types
  • Object Types
  • Function Types

Examples

function randomBooleanGenerator() {
    return Math.random < 0.5;
};

const vehicle = {
    make: 'Toyota',
    model: 'Tacoma',
    year: 2014,
}

console.log(`The vehicle weighs : ${vehicle.weight}`);
Typescript would flag both of these as problems out of the box.

Compiling

Typescript is compiled to Javascript with the tsc compiler.

// testfile.ts
console.log('Hi there');

// tsc testfile.ts -> testfile.js

Compiler Options

Typescript defaults to targetting Javascript ES3 but accepts options that let us change that :

  • --target es2015
  • --strictNullChecks - Require explicit null checking
  • --noImplicitAny - Error when an implicit "any" type is found
  • --strict - Enable all strict type checking options

Types

Typescript has support for the Javascript primitive types of string, number, and boolean. It also supports arrays of a specific type.

let name: string = 'test';
const age: number = 25;
const hobbies: string[] = ['knitting', 'cycling', 'swimming'];

function doWork(what: string, howLong: number, why: string[]) {

}

Return Types

We can also define the exact type that a function will return by appending a colon and the type after the parameters :

function doWork(what: string): boolean {
    return false;
}

Object Types (Intro)

In addition to primitive types, Typescript also supports Object types with optional properties

function doWork(work: { what: string, when: number, why?: string[] }): boolean {
    return false;
}
Note : If an optional property is used it must be checked for undefined.

Union Types

Union types which allow the choice of multiple types for a property. However, any use of the variable must either be "narrowed" or be a use that applies to all specified types.

function print(id: string | number) {
    console.log(id);
}

Code Narrowing

You can "narrow" the code to allow for flexible usage by doing explicit type checking. This allows you to use properties/functions of a specific type that aren't available on the other while maintaining Typescript compliance.

function print(list: string | string[]) {
    console.log(list.length); // Valid
    console.log(list.substr(0, 3)); // Invalid
    
    if (typeof list === 'string') {
        console.log(list.substr(0, 3)); // Valid (narrowed)
    }
}

Complex Types / Type Aliases

Rather than define the type of every property, of every object, every time it is used, you can define a complex type to describe it once and then reuse it. This is done with the type keyword.

type Vehicle = {
    make: string,
    model: string,
    year: number,
};

function doSomething(vehicle: Vehicle) {

}

Interfaces

Javascript doesn't support interfaces, though we looked at a way to get by with a class. Typescript does :

interface Vehicle = {
    make: string,
    model: string,
    year: number,
};

function doSomething(vehicle: Vehicle) {

}
* Interfaces can be extended, types are constant.

Type Assertions

There are times when you will use a Javascript function or property that Typescript has no way to know the type of. In these cases, you can specify it manually with an assertion. 

const result = someModule.doSomething() as string;

Literal Types

Literal types allow us to define not only the type of a variable, but also the acceptable values. Similar to enums in JSON Schema.

function paint(color: 'red' | 'blue' | 'green') {
}

type Vehicle = {
    make: string,
    model: string,
    year: number,
  };
  
function doSomething(vehicle: Vehicle | 'none') {
}

Function Type Signatures

Typescript also allows us to define the format and types of functions when they are passed as parameters.

function output(fn: (a: string) => void) {
    fn("Hello, World");
}

Construct Signatures

Similar to function type signatures, Typescript allows us to provide the definition of constructors invoked with the new keyword. 

type myConstructor = {
    new (param: string): object;
};

type myConstructorOrInvocation = {
    new (param: string): object;
    (param: number): number;
};

Generics

When a function can handle multiple different types and the output type will relate to the input type, we can use generics to document this.

function myFunction(param: Type[]): Type {
    return param[0];
}

Optional Function Parameters

Typescript can enforce proper function parameter calls to ensure that we are calling the correct function with the correct number of parameters. To satisfy this constraint while allowing optional parameters we can mark them as "optional" with a question mark.

function doWork(p1: string, p2?: number) {
    // p1 is required, p2 is optional
}

doWork('hi'); // Valid
doWork('hi', 2); // Valid
doWork(); // Invalid

Function Overloads

We can call Javascript functions in many flexible ways. Typescript limits that somewhat to conform to type definitions. 

function makeDate(timestamp: number): Date;
function makeDate(m: number, d: number, y: number): Date;
function makeDate(mOrTimestamp: number, d?: number, y?: number): Date {
    if (d !== undefined && y !== undefined) {
        return new Date(y, mOrTimestamp, d);  
    } else {
        return new Date(mOrTimestamp);  
    }
}
const d1 = makeDate(12345678); // Valid
const d2 = makeDate(5, 5, 5); // Valid
const d3 = makeDate(1, 3); // Invalid

Function Overloads / Implementation Signature

The final signature is the "implementation signature" and must be compatible with the overloads but is not itself an overload.

Special (Function) Types

  • void - When a function does not return a value. Inferred when there is no return statement.
  • object - Any value that is not a primitive (note the case)
  • unknown- Similar to any but can never be used.
  • never - A type that can never happen, and therefore can never be used.
  • function - To define a Function type but returns an any

Rest Parameters

If you remember rest parameters from Javascript, we can define them similarly in Typescript. Since they are transformed into an array, we can prefix them with the ... rest operator and type them as an array (of a single type).

function doWork(param1: string, ...param2: number[]) {
    // ...
}

doWork('test', 1, 2, 3, 4); // param1 = 'test', param2 = [1, 2, 3, 4]

Spread Syntax

Spread syntax can be used in the same way. Using the same example as above we can do :

function doWork(param1: string, ...param2: number[]) {
    // ...
}

const numArray = [1, 2, 3, 4];
doWork('test', ...numArray); // param1 = 'test', param2 = [1, 2, 3, 4]

Parameter Destructuring

Of course, parameter destructuring is available as well but with a slightly different syntax. We simply need to define the destructuring, then define the type description afterwards

function logger({ p1, p2, p3 } : { p1: string, p2: string, p3: string }) {
    console.log(`${p1} - ${p2} - ${p3}`);
}

const param = {p1: 'a', p2: 'b', p3: 'c'};
logger(param);

Readonly

Properties can be marked as readonly to denote that we can read from them, but never write to them.

interface Vehicle {
    make: string,
    model: string,
    year: number,
    readonly recalled: boolean,
}

function doWork(vehicle: Vehicle) {
    console.log(`vehicle recalled = ${vehicle.recalled}`); // Valid
    vehicle.recalled = false; // Invalid
}
