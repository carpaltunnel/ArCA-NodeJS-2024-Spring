API Documentation





While not terribly interesting, documentation is an essential part of any project.

Open API

The OpenAPI Specification defines how we'll describe and document our API while Swagger provides a convenient UI/interpreter that is easy to use and navigate. It supports both JSON and YAML type definitions. We'll stick to JSON since we're using it everywhere else but will show a YAML comparison because the YAML definition is a bit more brief and easy to use

Required Root Level Fields

The OpenAPI spec defines the following required top level fields

  1. openapi - The OpenAPI spec version the document adheres to
  2. info - Metadata about the API (title, description, version)
  3. paths - Available paths and operations in the API

Optional Root Level Fields

The following top level fields are optional :

  1. servers - Array of server objects that define where the API is found
  2. components - Defines the schemas and other reusables in the spec
  3. security - Definition of the security mechanisms used in the API
  4. tags - Object that defines the tags used in the API with descriptions
  5. externalDocs - Additional external documentation

info

The info object supports the following properties where required fields are marked with an asterisk (*)

  1. *title - Title of the API
  2. description - Short description of the API
  3. termsOfService - URL (only a URL) to the terms of service
  4. contact - Contact information for the API company/team
  5. license - API license
  6. *version - API version. Not the OpenAPI version, the version of the API being described

components

The components section contains reusable blocks of definitions for (re)use elsewhere in our spec. This is a great place to store things like schemas, requestBodies, responses, and parameters that will be used multiple times in the spec.

  • schemas
  • responses
  • parameters
  • examples
  • requestBodies
  • headers

components Continued

  • securitySchemes
  • links
  • callbacks

schemas

The schema object defines data types in your spec. This is an extended subset of the JSON Schema that we have looked at previously and functions generally the same. The following are taken directly from the JSON Schema spec

  • title
  • multipleOf
  • maximum
  • exclusiveMaximum
  • minimum
  • exclusiveMinimum

schemas Continued

  • maxLength
  • minLength
  • pattern (regex)
  • maxItems
  • minItems
  • uniqueItems

schemas Continued (again)

  • maxProperties
  • minProperties
  • required
  • enum

schemas : Non-JSON Schema Standard

These properties are taken from the JSON Schema spec but have a slightly different definition.

  • type - String, only a single value is supported
  • allOf - Inline or referenced schema must be an OpenAPI Schema object, not a JSON Schema
  • oneOf - Same as above
  • anyOf - Same as above
  • not - Same as above

schemas: Non-JSON Schema, continued

  • items - Must be an object, not an array. Schema must be an OpenAPI Schema object, not a JSON Schema
  • properties - Schema must be an OpenAPI Schema object, not a JSON Schema
  • additionalProperties - Boolean or object. If object, must be an OpenAPI schema object
  • description - May contain CommonMark syntax
  • format - OpenAPI offers a few extra formats compared to JSON schema
  • default - Must match the type of this schema

headers

The header object has the same structure as the Parameters object with minor changes

  • name must not be specified, it is taken from the headers key
  • in must not be specified since we're explicitly defining headers so... they'll be in the header.
  • All traits affected by location must adhere to a header

requestBodies

The request body object defines the possible request body types and contents that can be supplied to an endpoint. The request body object itself is quite simple with all the body definition being done in the content nested Media Type object. The base properties are :

  • description
  • required
  • content - Map of media type keys where the value describes it.

Media-Types (content)

The media type object, defined by the content keys, provides a schema and examples that define the associated request body. Available properties are :

  • schema - Obviously the most important, this is a Schema type object that defines the body
  • example - A request body example that adheres to the schema
  • examples - Same as example, but an array with multiple examples
  • encoding - Object to define the various encoding types

paths

We discussed the paths object earlier. Here is a very simple example, but we'll implement a full one shortly.

{
    "paths": {
        "/widgets/{id}": {
            "get": {},
            "put": {},
            "delete": {}
        }
}
Use everything so far to build a monolithic OpenAPI doc.

References

References allow reuse of definitions to avoid code duplication with the $ref keyword

"/widgets/{id}": {
        "get": {
            "responses": {
                "200": {
                    "description": "Get a widget by ID",
                    "content": {
                        "application/json": {
                            "schema": {
                                "$ref": "#/components/schemas/widget"
                            }
                        }
                    }
                }
    ...

Use References



Now, let's rework our API spec to use references and avoid all the schema code duplication.

Multi-File Definitions

We can use $ref to reference other files. This allows us to make our definition files smaller and easier to manage by splitting the large parts, like schemas, into dedicated files.

"components": {
    "schemas": {
        "widget": {
            "$ref": "schemas/widget.json#/components/schemas/widget"
        }
    }
}

Testing and Development


The Swagger Editor is great for easy development and testing of single file specs but it can't natively handle specifications that are split across multiple files. The Swagger UI can, but only if those files are publicly available on the internet. You could host it on Github or some other public location, but for local-only development we need a few more steps.

Local Swagger UI

Download the Swagger UI project and install http-server globally (NPM)

// Swagger UI directory
npm start
cd ./my/openapi/directory
http-server --cors -a 127.0.0.1
// Open browser to  http://127.0.0.1:3002
// Enter JSON spec file of http://127.0.0.1:8080/your-spec-file.json
